Why Nostr, not libp2p

The chaos protocol picks discovery and transport as separate problems and gives each one the right tool. This is the short version of why.

chaos · architecture decision record · D1 from PRD.md §12

The core insight

Discovery and transport look like the same problem, so most projects pick one tool to do both. They’re not. chaos splits them.

Discovery layer

Lightweight signed metadata. Tag-indexed pub/sub. Federated relays. Sovereign keys.

libp2p tries to solve both problems with one tool. It does neither in the shape we need.

The five reasons, one card each

01

Tag-indexed subscription is a primitive

Nostr relays are built to index events by tags. With libp2p we’d be reinventing the relay we already get for free.

02

The federation already runs

Modes B and C of our deployment plan assume independent community relays. With Nostr those exist today and our smoke test publishes to them right now.

03

Sovereign identity by design

Nostr was built around “users own their keys.” That maps 1:1 to AGENTS.md Rule 3. libp2p has peer IDs — not the same thing.

04

Operators can run a relay on a $5 VPS

strfry is one Go binary. The easier we make running a relay, the more independent operators federate, the more decentralised the network actually is in practice.

05

WebSocket on 443 just works, everywhere

Through firewalls, NATs, captive portals, corporate proxies, mobile hotspots. For a protocol that has to run on a developer’s laptop on coffee-shop wifi — which is exactly the smoke test we’re running now — that matters enormously.

What libp2p IS good at — and why we still defer it

Deferred,
not excluded

True multi-party gossip (gossipsub)

If chaos ever needs many-to-many consensus — a swarm of agents collaboratively deciding something — gossipsub is genuinely the right primitive. We’re explicit about this in the protocol doc:

“True multi-party consensus is not in scope for v1–v3. If we ever need it, that’s a separate layer above MCP using libp2p / gossipsub. The current design doesn’t preclude this — it just doesn’t ship it.”
— PROTOCOL.md, line 349

The shape we ended up with

Hermes Agent runtime · plugin host · skill layer · MCP client runtime
MCP Direct agent-to-agent transport · ImageContent · EmbeddedResource delivery
Nostr NIP-99 listings · tag-indexed pub/sub · federated relays · sovereign keys discovery

Nostr discovers. MCP delivers. Hermes runs it.
Composed, the system is smaller and more decentralised than either tool alone would produce.
