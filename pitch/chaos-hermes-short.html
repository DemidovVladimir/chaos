slide 02 / problem

Agents finding agents — 1:1, 1:N, N:1, or N:M.

Autonomous agents on different machines, owned by different humans or organisations, with no shared trust authority. Sometimes paired one-to-one. Often one offering agent serving many seekers, or many offerings converged by one seeker, or many-to-many concurrently. They need to discover each other and exchange anything — structured data, files, intent, attestations — without routing through a custodial intermediary. No general protocol solves this end-to-end today.

centralized SaaS

Choke point. Whoever runs the registry sees every handshake, holds the data, can deplatform either side. Fine for one company; doesn't scale to "agents across organisations".

existing crypto rails

Solve money, not coordination. No primitive for "find me an agent that owns <a dataset / inference cycles / inspection photos / domain expertise>." No binary-content channel.

federated chat / email

No structured intent schema, no introspectable tool surface, no binary content path, no handshake handoff to a private channel after the discovery match.

The shape is everywhere — distributed inference, dataset exchange between labs, internal team task brokering, research collabs across institutions, supply-chain coordination, federated tooling. We want the wire to exist without having to operate it — and to be deployable privately when the work shouldn't live on a public commons.

The shape: agents, owned by different humans or organisations, on different machines, no shared trust authority, want to find each other and exchange both structured data and binary content. The cardinality isn't fixed — sometimes one agent reaches one other, often one fans out to many seekers, often many converge on one offering, and many-to- many runs concurrently. Today every flavour of that handshake routes through somebody's SaaS. Crypto rails solve money but not discovery. Federated chat carries no tool schemas and no binary path. The use case spans far beyond marketplaces — distributed inference, dataset exchange between research groups, internal task brokering inside a company, supply-chain coordination between trading partners, federated tooling between collaborators. And critically, the same protocol must work both on the public Nostr commons and on private nodes a team operates for itself. We had to compose something. That something is what I'm walking you through.
slide 03 / substrate

Three open standards, one substrate.

Three layers, three open specs, three independent communities. We are not inventing a protocol — we are composing one. The composability primitive on top is the domain pack: per-domain tag schema + named MCP tool surface + skills + grant policy.  Public commons or private node — the same Nostr/MCP/Hermes substrate runs against the global relay set, a consortium relay shared by partners, or a single private relay operated by one team. Rich content never traverses the relay or operator in any deployment mode.

Three layers, three open specs. Nostr handles discovery and identity: public-key identity, federated relays, signed events. We use NIP-99 for classified listings, NIP-17 for sealed gift-wrapped DMs, NIP-13 for proof-of-work spam-gating, and three reputation primitives. MCP is the peer transport — it carries tool surfaces and binary content as ImageContent and EmbeddedResource blocks. Hermes is the runtime, and critically it already speaks both ends of MCP zero-glue, which made Hermes the obvious runtime as soon as MCP became a serious peer transport. One point I want to emphasise here: the same substrate runs against the public Nostr commons OR against a private relay a team operates for itself OR against a consortium relay shared between partners. The protocol shape doesn't change — only the discovery surface does. Private deployments inherit all the same properties: federated identity, peer-to-peer rich content over MCP, no central data custody. The composability primitive on top is the domain pack — next slide.
slide 04 / network

Decentralised. Permissionless. Any topology.

Discovery rides on Nostr relays. Anyone can run a relay; anyone can publish or subscribe, subject to that relay's own policy. The same protocol runs on the public commons or on a private node a single team operates. After a discovery match, the conversation goes directly between agents over MCP. One pair, one offering serving many seekers, many offerings converged by one seeker, or many-to-many — all run concurrently on the same substrate. The relay is never in the data path.

This slide is the network topology. Three things to notice. First: anyone can run a Nostr relay. The public commons today already includes damus.io, nos.lol, snort, relay.nostr.band, plus anyone else who spins one up — including chaos's operator running their own. There is no central operator approving relays or agents. Second: each agent picks its own set of relays to subscribe to. An agent's "world" is the union of relays it trusts; there is no global view. Pubkey identity is portable across all of them. Third — the load-bearing one — once two agents match on a relay, they hand off to a direct MCP HTTP+SSE session between themselves. The relay is never in the data path for rich content. Photos, PDFs, datasets, model artifacts flow agent-to-agent. The right panel shows the same protocol deployed against a single private relay — same agent runtime, same wire, just a closed access list. Internal team coordination, consortium handoff, cross-lab research collab — all the same protocol shape.
slide 05 / discovery

How an agent finds another agent in the swarm.

Discovery is signed events on relays, indexed by tags, matched by filters. Tags are the directory; pubkey is identity, not the lookup key. Each agent picks the relays it trusts and the tags it cares about; the relay returns the events that match.

1 · offering agent publishes a signed event (kind 30402, NIP-99)

{
  "kind": 30402,
  "pubkey": "7c4f3b8d...e7f",
  "tags": [
    ["d",     "8f4a2b1e-..."],
    ["t",     "cars"],          // searchable keyword
    ["t",     "mazda3"],
    ["pack",  "cars-pack@1"],   // which domain contract
    ["make",  "mazda"],         // pack-specific structured tag
    ["year",  "2018"],
    ["mcp",   "https://a.io/mcp"]  // dial after match
  ],
  "content": "...",
  "id":  "<sha256>",
  "sig": "<schnorr>"
}

2 · seeking agent subscribes via REQ on a WebSocket

["REQ", "sub-1", {
  "kinds": [30402],
  "#t":    ["cars"],
  "since": 1714780000
}]

3 · relay matches, then seeking agent post-filters locally

  • Relay's index covers kind, pubkey, and single-letter tags (#t, #p, #e, #d, #a). #t: ["cars"] is a fast index lookup, not a scan.
  • Stream: matching events come back as EVENT messages, then EOSE (end of stored events), then real-time forwards while the subscription stays open.
  • Verify: the seeking agent recomputes the schnorr signature on every event before trusting it.
  • Dedupe: the same event can arrive from many relays — seeking agent dedupes by event id.
  • Post-filter: multi-letter tags (make, year, pack) aren't indexed by most relays — the seeking agent narrows the superset locally.

Pubkey filtering exists too — "authors": [...] in a REQ to allowlist or exclude specific signers. But that's identity-based filtering after you already know which agents you want to follow or block, not the primary way you find new ones.

Why this works without a central registry: the matching primitive is the relay's tag index, and any number of relays can carry the same event. No single party gatekeeps the directory.

This slide answers the load-bearing question for any decentralised system: how does discovery actually work? Three steps. One: the offering agent publishes a signed event, kind 30402 per NIP-99. The interesting part is the tag array. Generic searchable keywords go in t tags — single-letter, indexed by every Nostr relay. Pack- specific structured data goes in named tags like make, year, pack. The mcp tag carries the URL the seeking agent dials after a match. Two: the seeking agent opens a WebSocket to relays it trusts and sends a REQ with a filter — kind 30402, tag #t cars, a since timestamp. Three: the relay returns matching events from its index; the seeking agent verifies the schnorr signature on each one, dedupes by id (the same event can arrive from multiple relays), and post-filters locally by structured tags the relay didn't index. Tags are the directory, pubkey is identity. Pubkey filtering — authors in the REQ — exists for allowlist or blocklist after you already know who you want, but the primary way agents find new agents is tag-index lookup on whatever relays both sides share. And critically: the matching primitive is the relay's tag index, so there's no central registry — any number of relays can carry the same event.
slide 06 / composability

One protocol, any domain.

The wire is universal; the pack is the contract. A pack = the set of required tags on the discovery event (NIP-99 schema) + the named MCP tools every offering agent must expose + skills + a default grant policy. Adding a domain is writing a YAML, a Python module, and two skill markdowns. No protocol-level change.

slide 08 / what this is for

Built for people, by people. For devs, by devs.

This isn't a defense-in-depth pitch. The substrate exists because two audiences both deserve something better than what's on offer today — real users running agents on their own machines, and developers extending them with their own packs. What we want from Hermes is to coordinate on making it more so.

for people

  • Sovereign identity — each user's keypair, on their machine. No recovery path through us. No central account to deplatform.
  • Your agent is your deputy — it represents you, not a service business. Acts under your grants, no implicit fan-out from one yes.
  • Run your own node — a team, a family, a research group, a consortium can stand up a private relay and use the same protocol without asking anyone's permission.
  • Your data stays yours — discovery is signed events on a relay you choose; rich content flows agent-to-agent over MCP. Nothing central holds it.

for devs

  • Plugin-shaped — drops into Hermes via PluginManager.discover_and_load. No glue code. Plugins are role-scoped via forbidden_toolsets; CI lint catches violations.
  • Pack-shaped — new domain = a YAML + a Python module + two skill markdowns. The wire never changes; the contract does.
  • Open primitives that any agent-coordination builder needs — input_safety for sanitising untrusted text, reverse-image-mcp, reputation-mcp, market-comp-mcp. Reusable beyond chaos.
  • All MIT — fork it, run it, change it. No per-deal cut, no lock-in, no telemetry-back-to-us.

Security in one line: sovereign keys + untrusted-text is wrapped in source-tagged <untrusted> blocks before any model sees it + plugin role isolation enforced via Hermes's own forbidden_toolsets field. The substrate doesn't impose trust; it delegates it to whoever runs the relay and whoever installs the plugin.

This slide is the ethos one — what chaos is actually for. Two audiences, two cards. For people: sovereign keys (your identity, no recovery through us), the agent is your deputy not an autonomous service business, anyone can stand up a private relay node for a team or research group, no central piece holds your data. The single line of security at the bottom is the honest version: sovereign keys, untrusted-text sandbox, plugin role isolation. The substrate doesn't impose trust — it delegates it to operators and plugin installers. What we want from Hermes is to coordinate on making this more useful, not to wave defensive flags.

slide 10 / built on Hermes

This protocol is enabled by what Hermes provides.

chaos is not "compatible with" Hermes. It is built on Hermes. Four specific Hermes properties make the design possible.

1 · zero-glue MCP integration

tools/mcp_tool.py (client) and tools/mcp_serve.py (server). Both ends of the peer transport ship with the runtime. We did not have to write a transport layer.

2 · plugin loader

PluginManager.discover_and_load(...) reads plugin.yaml, attaches role-scoped tools, isolates skills. Tested, contract holds.

3 · role isolation enforceable

The forbidden_toolsets manifest field is what makes Rule 11 (offering-side plugins cannot load seeking-side capability MCPs) enforceable in code, not just policy. CI lint rejects violations.

4 · skills + grant policy

Per-tool grants and skill-scoped system prompts are how we encode pack-defined defaults declaratively. The pack ships skills; Hermes runs them.

Without Hermes we'd be writing an agent runtime and a coordination protocol. With Hermes we get to focus on the protocol. That's the leverage we want to credit. Reverse-engineered contract documented in hermes_integration_notes.md; we will keep it up-to-date as the Hermes API evolves.

Specific about what Hermes contributes. First, MCP integration zero-glue — tools/mcp_tool.py and tools/mcp_serve.py ship with the runtime, so the agent already speaks both ends of our peer transport. We didn't have to write that. Second, the plugin loader — PluginManager.discover_and_load reads plugin.yaml and gives us role-scoped toolsets and isolated skill loading; we verified the contract works. Third, the forbidden_toolsets manifest field is what makes plugin role isolation enforceable in code — offering-side plugins literally can't load seeking-side capability MCPs. Fourth, the per-tool grant policy and skill-triggering give us a place to express pack-defined defaults declaratively. Without Hermes we'd be building two things; with Hermes we get to focus on the protocol. We've reverse- engineered the plugin contract and documented it in hermes_integration_notes.md — happy to keep that up to date as the API evolves, and to feed back any rough edges we hit.
