chaos — defense in depth

Click any layer to see what it catches and where it's enforced. The agents in this product are exposed to untrusted text on every event they read. Sanitizing input is the first line of defense; toolset narrowing, approval gates, and container isolation are the rest.

3Toolset narrowing
no terminal · no execute_code · no delegation · no web
4User approval gates

5Per-ask grant policy
seller decides what to share, per buyer, per ask
6Pre-share photo check
reverse-image-mcp before any MCP tool-result delivery
7Container isolation
read-only root · tmpfs /tmp · egress allowlist
8Output redaction
API keys, PII patterns scrubbed at display

Click a layer

Each layer is independent. The architecture is "defense in depth" — no single layer is sufficient, and a successful injection has to bypass every applicable one.

Layers 1, 2, 5, 6 apply to seller and buyer agents on every event read or sent. Layer 3 is set per-skill at install time. Layers 4, 7, 8 are operator-configured at deployment.
