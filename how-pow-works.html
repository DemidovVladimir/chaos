How NIP-13 PoW protects the relay

A short walkthrough of the spam gate — what it is, how it’s computed, why 20 bits, and what attacks it does and doesn’t prevent.

chaos · spec: NIP-13 · default 20 bits per AGENTS.md rule 10
TL;DR

Each listing publish must include a nonce chosen so the event’s SHA-256 id has at least 20 leading zero bits. Finding such a nonce takes ~1 million SHA-256 attempts on average — a fraction of a second for an honest publisher, hours for a spammer trying to flood. The relay rejects events without enough leading zeros.

The mining loop, in five steps

1
Build the unsigned event kind=30402, the listing tags, the offering pubkey, created_at, content
build_event()
│
2
Add a nonce tag ["nonce", "", "20"] — the second field is the iterating value, the third declares the target
tags.append
│
3
Compute the canonical event id SHA-256 of the JSON-serialized [0, pubkey, created_at, kind, tags, content]
SHA-256
│
4
Count leading zero bits in the id If ≥ 20, mining is done. If < 20, increment the nonce value and loop.
leading_zeros
↑ loop back to step 3 with nonce + 1 (~1M iterations on average for 20 bits)
5
Sign and publish Schnorr-sign the final id, ship it to the relay over WebSocket
sign() · publish()

How “20 bits” is measured

Without PoW
your smoke-test event
f8711652f17d… starts with f — that’s 1111 in binary — 0 leading zero bits
20-bit PoW (min)
000007b8a4f3e2c1d0… five hex zeros = 20 zero bits. The next char must be 0–7 (top bit is 0). ≥ 20 leading zero bits ✓
24-bit PoW
000000a7b8c2d1e0… six hex zeros = 24 zero bits. 16× harder to find than 20 bits.

Quick mental model: each hex character is 4 bits. 20 bits ≈ 5 hex zeros. Each additional bit doubles the expected work to find a valid id. So 20 → 21 bits is 2× cost, 20 → 24 is 16×, 20 → 28 is 256×.

The economics — why 20 bits

Bits Avg attempts ~Time per event
(modern laptop, Python)		 Honest user
(1 listing)		 Spammer
(1000 listings)
8 ~256 < 1 ms imperceptible trivial
16 ~65,536 ~50 ms imperceptible ~50 sec
20 ~1,048,576 ~500 ms – 2 s tolerable ~10 min – 30 min
24 ~16.7 M ~8 – 30 s annoying ~2 – 8 hours
28 ~268 M ~2 – 8 min painful ~33 – 130 hours

20 bits is the chosen sweet spot: barely noticeable for an honest publisher posting one or two listings a week, but enough to make a 1000-listing flood take 10+ minutes per pubkey of pure CPU work. Higher bits (24, 28) push spammer cost up but also start to hurt honest publishers on weak hardware and mobile devices.

What it prevents — and what it doesn’t

Stops

  • ✓
    Single-key flood spam Each event costs CPU. Linear in volume.
  • ✓
    Cheap-key sybil spam Generating a new pubkey is free, but each event still requires its own PoW.
  • ✓
    Relay DB-fill via volume Spammer must spend ≥ 1M hashes per row written.
  • ✓
    Race-to-publish duplicates Forging tens of variants of the same listing is now expensive.

Doesn’t stop

  • ✗
    Botnet-distributed spam If 10k bots each pay PoW, the network sees high-volume valid traffic.
  • ✗
    Network-layer DDoS TCP flood, slowloris, BGP attacks — that’s a CDN / firewall job.
  • ✗
    Subscription flooding PoW is on publish. REQ filters need separate per-connection rate limits.
  • ✗
    Smart low-volume spam An attacker posting 10 fake listings a day pays ~5 sec total. PoW is a volume gate.

PoW is one of four layered defenses

L1
NIP-13 Proof of Work Cost-per-publish gate. The thing this doc is about.
computational
L2
Paid relays Some operators charge a one-time pubkey-registration fee. Real money, not just CPU.
financial
L3
NIP-51 mute lists Each user’s agent locally filters out pubkeys it doesn’t want to see. No platform-level censorship.
user-side
L4
Pubkey reputation history Cumulative trust signal: NIP-58 badges, peer attestations, NIP-02 web of trust. Slow attackers can’t fake age.
temporal

None of these alone is a gatekeeper. PoW makes spam cost CPU; paid relays make it cost money; mute lists let users opt out; reputation makes brand-new pubkeys low-trust by default. An attacker has to defeat all four to flood the network meaningfully.

Where it actually lives in the chaos repo

The current build is half-implemented

The 20-bit floor is a rule in AGENTS.md. The implementation status varies by component:

mvp/agent_offering.py
No PoW code at all — the publish flow goes straight from build → sign → ship. Documented in mvp/README.md line 253: “The MVP skips NIP-13.”
SKIPS PoW
agent/src/chaos_agent/
publish.py
Function signatures defined: build_event, mine_pow, publish. All currently raise NotImplementedError. DEFAULT_POW_BITS = 20 is set.
STUB
operator/cars/
(strfry config)
The Mode A relay deployment configures strfry to reject kind-30402 events with fewer than 20 leading zero bits. Choke point at the relay, not the publisher.
ENFORCED
public Nostr commons
(damus, nos.lol, …)
Most public relays don’t enforce PoW for arbitrary kinds. That’s why your smoke-test event id f8711652f17d… (zero leading zeros) was accepted — they’re lenient by default.
NOT ENFORCED

The takeaway: the protocol design mandates 20-bit PoW, the operator-run relay enforces it, but the MVP scripts and the public relays we’re testing against don’t. Closing this gap is on the “queued” list — implementing mine_pow in agent/src/chaos_agent/publish.py is the next concrete step before the production agent migrates off the MVP scripts.
